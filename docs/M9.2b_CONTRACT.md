# M9.2b Contract — "Two-phase goal (§L) + describe-first (§B) + rich RunConfig" (frozen 2026-06-27)

> 🌐 **Русский** (основная версия) · [English](M9.2b_CONTRACT.en.md)

Завершает conversational-авторинг эпика **M9** (M9_CONTRACT §B/§L, ADR-022). M9.2a дал
**однопроходный** `GoalPlanner` (выбирает следующего реального кандидата на **текущей** странице).
M9.2b даёт полный сценарий §L/§B + богатый RunConfig.

**Срез M9.2b (этот цикл):**
- **§L двухфазный:** goal-режим сначала делает **полный explore** (карта всех страниц/элементов +
  обнаружение проблем), потом строит сценарий под цель по **полной карте** — находит И рабочие
  элементы, И проблемы, и авторит кросс-страничный бизнес-процесс (логин→биллинг).
- **§B describe-first:** описать флоу словами → LLM **черновик** → explore **сверяет** (reconcile)
  черновик с реальной картой (привязывает каждый шаг к реальному элементу или помечает unmatched).
- **Богатый RunConfig:** декларативная `auth:`-секция + именованные `scenarios:` + селектор `--scenario`
  поверх минимального RunConfig из M9.2a.

Закрывает **GAP-M9-02** (goal + describe + двухфазность) и **GAP-M9-09** (богатый RunConfig).

**Несущая предпосылка:** сегодня `grounding` (привязка) — **только по кнопкам**
(`_buttons_from_interactives`). Логин/форма/биллинг требуют input/select, поэтому карта должна сперва
**обобщиться** на роли textbox/combobox/checkbox/link — это и есть основная архитектурная работа.

Проверка — **offline-first**: многостраничный `FakeEx` + `FakeBackend`; живой goal/describe-прогон — по «go».

## §1. Общая машина (goal и describe — одна машина, две головы фазы-2)

Обе ветки = **фаза 1: полный детерминированный explore (HeuristicPlanner)**, строящий персистентную
**карту сайта**, затем **фаза 2: один LLM-вызов**, дающий привязанную к реальным элементам
воспроизводимую последовательность шагов. LLM **никогда не ведёт walk** (токены тратятся только в
фазе-2) → обнаружение дёшево и детерминировано, best-effort изолирован в авторском хвосте.

## §2. Карта сайта (обобщение за пределы кнопок)

Новое поле состояния `site_map: {page_path -> [element]}`. Элемент переиспользует существующий
дескриптор (`semantic_id`/role/name/testid/locator/alternatives) + `page`. `_buttons_from_interactives`
→ `_elements_from_interactives` (button **+ input/select/link**, та же дисциплина `alternatives`).
**Покрытие остаётся по кнопкам** (не трогаем convergence/`plan_hash` чистого explore); карта — superset
только для авторинга, накапливается в узле `ground` (dedup по `semantic_id`).

## §3. Двухфазный goal — `GoalPlanner.build_scenario` (one-shot)

`GoalPlanner.build_scenario(flat_map, goal)` — **один** бюджетируемый LLM-вызов (не per-step): LLM
возвращает упорядоченный список `{ref: <semantic_id>, verb, value?}`. **`grounding`:** `ref`, которого
нет в карте — **или** `verb` вне `{click,fill,type,select,press,assert}` — отбрасывается в `unmatched`
(никогда не фабрикуется). **Кросс-страничные `navigate`
синтезируются в КОДЕ** из ключа `page` элемента (реальный URL, не free-text LLM). Per-step `propose`
(M9.2a) остаётся в коде для будущего live/co-pilot пути (M9.4), но больше **не** провод walk-планировщика.

## §4. describe-first — `DescribePlanner.draft` + детерминированный `reconcile`

`DescribePlanner(description, backend).draft()` — один LLM-вызов → намеренные шаги
`{verb, intent, hypothesized_target}` (ungrounded by design). `reconcile(draft, site_map)` —
**детерминированный код-матчер** против карты (нормализованные role+name точно → text → page-scoped) →
привязка к реальному элементу **или** `unmatched`. **Консервативно: никогда не угадывает** (неоднозначность
→ unmatched). Сильнее `grounding`, чем goal: LLM вообще не выбирает селектор/индекс.

## §5. Артефакты + инвариант детерминизма

- **`plan.json`** (walk+scenario, единый `canonical_plan_hash`, существующий путь) **+** standalone
  **`scenario.json`** (чистый воспроизводимый бизнес-процесс — то, что гоняет CI). Шаги несут `phase:
  explore|scenario`. describe дополнительно пишет **`reconcile-report.json`**.
- **Инвариант детерминизма:** каждый авторский шаг несёт **полный привязанный `locator`+`alternatives`,
  скопированный из элемента карты**, и **ни одно недетерминированное поле** (LLM `reason`/score) не
  попадает в dict шага (`canonical_plan_hash` хэширует всё) → замороженный план **replay'ится LLM-free
  детерминированно**. goal/describe — best-effort (как ADR-019), но реплей заморозки детерминирован.

## §6. Коды выхода

- describe с **любым** unmatched-шагом → **exit 1** (CI ловит «описанный флоу не существует»).
- goal/describe, давший **ноль** привязанных шагов → **exit 1**.
- `GOAL` И `DESCRIBE` заданы одновременно → **exit 3** (взаимоисключающие головы).
- Битый RunConfig → **exit 3** (как в M9.2a). Сюда же: неизвестный `--scenario` (нет такого имени в
  `scenarios:`) и entry с **обоими** `goal`+`describe` (XOR-валидация) → **exit 3** (не тихий неверный флоу).

## §7. Богатый RunConfig (декларативно → существующий env; адаптеры = M9.7)

```yaml
auth:                       # декларативно -> env M9.1 (НОВОГО runtime нет)
  storage_state: state.json #  -> STORAGE_STATE
  storage_state_save: out.json #  -> STORAGE_STATE_SAVE
  login_plan: login.json    #  -> PLAN_FILE (login-as-test)
  pw_no_trace: true         #  -> PW_NO_TRACE=1
scenarios:                  # именованный список; --scenario <name> выбирает ОДИН (§C: один режим на прогон)
  - {name: login, goal: "войти как admin"}
  - {name: signup, describe: "регистрация нового пользователя"}
```

`auth`/`scenarios` добавляются в allow-list + валидируются (битая вложенность → exit 3). `_apply_auth`
маппит в `STORAGE_STATE*`/`PLAN_FILE`/`PW_NO_TRACE`; `_apply_scenarios(selector)` берёт запись по
`--scenario` → `GOAL`/`DESCRIBE`. Приоритет **флаг>файл>дефолт** через `_overridable` по целевой env-переменной
(заранее экспортированный `STORAGE_STATE` бьёт файл). Бюджеты без изменений (vision → роль `heal`;
per-role endpoints = M9.7).

## ADR

- **ADR-028** (2026-06-27) — M9.2b двухфазный авторинг: goal/describe-режимы делают полный
  детерминированный heuristic-explore (карта сайта, обобщённая за пределы кнопок на input/select/link),
  затем **one-shot привязанную голову фазы-2** (`GoalPlanner.build_scenario` / `DescribePlanner.draft` +
  детерминированный `reconcile` в новом `brain/scenario.py`); кросс-страничные navigate синтезируются в
  коде; авторские шаги несут полный привязанный `locator`+`alternatives` (replay LLM-free детерминирован);
  `plan.json`(walk+scenario) + `scenario.json` + `reconcile-report.json`; богатый RunConfig (декларативные
  auth/scenarios + `--scenario`). **Заменяет** проводку M9.2a, где `GoalPlanner` был walk-планировщиком
  (`propose` сохранён для M9.4 live/co-pilot). describe-unmatched → exit 1; `GOAL`⊕`DESCRIBE` → exit 3.
  Accepted.
