## §1. `GoalPlanner` — a grounded goal-directed planner

Slots into the existing `Planner` seam (ADR-011, `brain/planner.py`) next to `HeuristicPlanner` /
`LLMPlanner`. The seam contract is unchanged:

```
propose(state: dict, candidates: list) -> {action: <candidate>|None, done: bool, reason: str, tokens: {...}|None}
```

**Grounding (ADR-022 — the key guarantee).** `GoalPlanner` receives the list of **real** candidates the
`plan` node assembled from the live element map (`{kind, semantic_id, role, name, target, intent, locator,
alternatives}`). The LLM picks a candidate **index** (like `LLMPlanner`), it does not author a selector.
`propose` **returns only `candidates[idx]` or `done`** — an invalid/OOB index ⇒ `done`, an action is
**never** fabricated that isn't on the map. So the LLM physically cannot hallucinate a locator.

**Difference from `LLMPlanner`:** the prompt is goal-directed — "pick the action that advances the
**goal**", with the goal text and progress; `done` = "goal achieved" (not just coverage). The prompt menu
is the same grounded `{i, kind, role, name, target}`.

**Graceful degradation (like `LLMPlanner`/ADR-019/021):** no goal · no backend (no key/SDK) · `plan`
budget exhausted → fall back to `HeuristicPlanner` (the backend is **not** called). So a missing key or an
exhausted budget never breaks a run; the heuristic is the deterministic anchor.

Constructor: `GoalPlanner(goal: str, backend=None)`; `name="goal"`, `model` = the backend model (a
transcript label) or the historical default.

## §2. Authoring mode — auto-default by `--goal` (no `--mode` collision)

**Name collision:** the existing `agentctl --mode` flag already maps to `RUN_MODE` (the **execution**
axis: explore / replay / baseline / …). So `--mode explore|goal|describe` from §C is **not** reused.

The authoring mode is selected like this (`make_planner(env)` in `brain/planner.py`):
- **Auto-default (§C):** `--goal "<text>"` set (env `GOAL`) → **goal-mode** (`GoalPlanner`); unset → explore.
- **Explicit:** `PLANNER=goal` (`--planner goal`) or RunConfig `mode: goal`.
- `PLANNER=llm` → `LLMPlanner`; otherwise → `HeuristicPlanner`.

No "complexity auto-detection": the signal is the presence of a goal — explicit and predictable.

## §3. RunConfig YAML (minimal) — `brain/runconfig.py`

Rich runs don't fit in flags (GAP-M9-09). The minimal RunConfig (M9.2a):

```yaml
mode: goal                 # explore | goal   (-> PLANNER)
goal: "log in as admin and open billing"
planner: goal              # heuristic | llm | goal (synonym for mode)
coverage_target: 0.85
max_steps: 40
plan_budget: 20000         # plan-role tokens  (-> budget)
heal_budget: 20000
```

- `load_run_config(path) -> dict` — parses the YAML (**pyyaml**, already in `.venv`; added to
  `pyproject`), validates against an allow-list of keys, **ignores unknown** keys (forward-compat for
  M9.2b auth/scenarios).
- `apply_run_config(cfg, env)` — merges into the environment. **Precedence: an explicit flag/`--goal` >
  the RunConfig file > defaults.** Activated when `RUN_CONFIG` is set (the `--run-config <path>` flag).
- **Real explicitness:** agentctl emits `SENTINEL_EXPLICIT` (the names of the flags the user actually
  passed, via `fs.Visit`) → the file never overrides an explicit flag, **even when its value == the
  default** (e.g. `--planner heuristic` is not flipped by a file `mode: goal`). Numeric keys are
  validated at load (a bad scalar → **exit 3**, not a silent exit 1); `mode`/`planner` are `PLANNER`
  aliases — a value conflict raises (exit 3).

## §4. Determinism

goal-mode is **best-effort (LLM), NOT `plan_hash`-stable** (like `LLMPlanner`, ADR-019): the model only
affects the explore artifact `plan.json`. **Replay of the frozen plan stays deterministic and LLM-free**
(the M3 trust layer is unchanged). `HeuristicPlanner` is the deterministic anchor + the degradation path;
golden baselines remain heuristic-only.

## ADR

- **ADR-027** (2026-06-26) — M9.2a GoalPlanner: a goal-directed grounded planner in the `Planner` seam
  (picks only real candidates by index → a selector hallucination is impossible, ADR-022); authoring mode
  by `--goal` presence (auto-default §C) + `PLANNER=goal`/RunConfig `mode`, **not** via `--mode` (reserved
  for `RUN_MODE`); a minimal RunConfig YAML (mode/goal/planner/budgets, precedence flag>file>default);
  goal-mode is best-effort (not `plan_hash`-stable, like ADR-019). Accepted. Deferred to M9.2b:
  describe-first, the two-phase explore-then-scenario, auth/scenarios in RunConfig.
