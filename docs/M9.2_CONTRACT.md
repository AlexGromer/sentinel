## §1. `GoalPlanner` — грауденный goal-directed планировщик

Встаёт в существующий шов `Planner` (ADR-011, `brain/planner.py`) рядом с `HeuristicPlanner` /
`LLMPlanner`. Контракт шва не меняется:

```
propose(state: dict, candidates: list) -> {action: <candidate>|None, done: bool, reason: str, tokens: {...}|None}
```

**Грауденность (ADR-022 — ключевая гарантия).** `GoalPlanner` получает список **реальных** кандидатов,
собранных узлом `plan` из живой карты элементов (`{kind, semantic_id, role, name, target, intent,
locator, alternatives}`). LLM выбирает **индекс** кандидата (как `LLMPlanner`), а не сочиняет селектор.
`propose` **возвращает только `candidates[idx]` или `done`** — невалидный/OOB-индекс ⇒ `done`, **никогда**
не фабрикуется действие, которого нет на карте. Поэтому LLM физически не может галлюцинировать локатор.

**Отличие от `LLMPlanner`:** промпт goal-directed — «выбери действие, продвигающее **цель**», с текстом
цели и прогрессом; `done` = «цель достигнута» (а не только покрытие). Промпт-меню — то же грауденное
`{i, kind, role, name, target}`.

**Graceful degradation (как `LLMPlanner`/ADR-019/021):** нет цели · нет backend (нет ключа/SDK) ·
бюджет `plan` исчерпан → откат на `HeuristicPlanner` (backend **не** вызывается). Так отсутствие ключа
или исчерпанный бюджет никогда не ломает прогон; heuristic — детерминированный якорь.

Конструктор: `GoalPlanner(goal: str, backend=None)`; `name="goal"`, `model` = модель backend (label
транскрипта) либо исторический дефолт.

## §2. Режим авторинга — авто-дефолт по `--goal` (без коллизии с `--mode`)

**Коллизия имён:** существующий флаг `agentctl --mode` уже маппится в `RUN_MODE` (ось **исполнения**:
explore / replay / baseline / …). Поэтому `--mode explore|goal|describe` из §C **не переиспользуется**.

Режим авторинга выбирается так (`make_planner(env)` в `brain/planner.py`):
- **Авто-дефолт (§C):** задан `--goal "<текст>"` (env `GOAL`) → **goal-режим** (`GoalPlanner`); не задан → explore.
- **Явно:** `PLANNER=goal` (флаг `--planner goal`) или RunConfig `mode: goal`.
- `PLANNER=llm` → `LLMPlanner`; иначе → `HeuristicPlanner`.

«Авто-определения сложности» нет: сигнал — наличие цели, явный и предсказуемый.

## §3. RunConfig YAML (минимальный) — `brain/runconfig.py`

Богатые прогоны не влезают во флаги (GAP-M9-09). Минимальный RunConfig (M9.2a):

```yaml
mode: goal                 # explore | goal   (-> PLANNER)
goal: "войти как admin и открыть биллинг"
planner: goal              # heuristic | llm | goal (синоним mode)
coverage_target: 0.85
max_steps: 40
plan_budget: 20000         # токены роли plan  (-> budget)
heal_budget: 20000
```

- `load_run_config(path) -> dict` — парсит YAML (**pyyaml**, уже в `.venv`; добавляем в `pyproject`),
  валидирует по allow-list ключей, **игнорирует неизвестные** (forward-compat для M9.2b auth/scenarios).
- `apply_run_config(cfg, env)` — мерж в окружение. **Приоритет: явный флаг/`--goal` > файл RunConfig >
  дефолты.** Активируется при заданном `RUN_CONFIG` (флаг `--run-config <path>`).
- **Честная явность:** agentctl передаёт `SENTINEL_EXPLICIT` (имена флагов, реально заданных
  пользователем, через `fs.Visit`) → файл не переопределяет явный флаг, **даже если его значение ==
  дефолт** (напр. `--planner heuristic` не флипается файловым `mode: goal`). Числовые ключи валидируются
  при загрузке (плохой скаляр → **exit 3**, не тихий exit 1); `mode`/`planner` — алиасы `PLANNER`,
  конфликт значений → ошибка (exit 3).

## §4. Детерминизм

goal-режим — **best-effort (LLM), НЕ `plan_hash`-стабилен** (как `LLMPlanner`, ADR-019): модель влияет
только на explore-артефакт `plan.json`. **Replay замороженного плана остаётся детерминированным и
LLM-free** (M3 trust layer не меняется). `HeuristicPlanner` — детерминированный якорь + путь деградации;
golden-baselines по-прежнему heuristic-only.

## ADR

- **ADR-027** (2026-06-26) — M9.2a GoalPlanner: goal-directed грауденный планировщик в шве `Planner`
  (выбор только из реальных кандидатов по индексу → невозможна галлюцинация селектора, ADR-022);

  Отложено в M9.2b: describe-first, двухфазный explore-then-scenario, auth/scenarios в RunConfig.
