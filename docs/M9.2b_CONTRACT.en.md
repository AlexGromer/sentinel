# M9.2b Contract — "Two-phase goal (§L) + describe-first (§B) + rich RunConfig" (frozen 2026-06-27)

> 🌐 [Русский](M9.2b_CONTRACT.md) (основная версия) · **English**

Completes the conversational authoring of epic **M9** (M9_CONTRACT §B/§L, ADR-022). M9.2a gave a
**single-pass** `GoalPlanner` (picks the next real candidate on the **current** page). M9.2b delivers
the full §L/§B scenario + a rich RunConfig.

**M9.2b slice (this cycle):**
- **§L two-phase:** goal-mode first runs a **full explore** (the whole site's element map + problem
  discovery), then builds the goal scenario over the **complete map** — finding both working elements and
  problems, and authoring a cross-page business process (login→billing).
- **§B describe-first:** describe a flow in words → LLM **draft** → explore **reconciles** the draft
  against the real map (binds each step to a real element, or flags it unmatched).
- **Rich RunConfig:** a declarative `auth:` section + named `scenarios:` + a `--scenario` selector, on top
  of M9.2a's minimal RunConfig.

Closes **GAP-M9-02** (goal + describe + two-phase) and **GAP-M9-09** (rich RunConfig).

**The load-bearing prerequisite:** today `grounding` is **button-only** (`_buttons_from_interactives`).
Login/form/billing needs inputs/selects, so the map must first **generalize** to textbox/combobox/
checkbox/link roles — this is the real architectural work.

Verification is **offline-first**: a multi-page `FakeEx` + `FakeBackend`; the live goal/describe run is on "go".

---

## §1. The shared machine (goal and describe — one machine, two phase-2 heads)

Both branches = **phase 1: a full deterministic explore (HeuristicPlanner)** that builds a persistent
**site map**, then **phase 2: one LLM call** that produces a grounded, replayable step sequence. The LLM
**never drives the walk** (tokens are spent only in phase 2) → discovery is cheap + deterministic, and the
best-effort behavior is isolated to the authored tail.

## §2. Site map (generalized beyond buttons)

A new state field `site_map: {page_path -> [element]}`. Each element reuses the existing descriptor
(`semantic_id`/role/name/testid/locator/alternatives) + `page`. `_buttons_from_interactives` →
`_elements_from_interactives` (button **+ input/select/link**, the same `alternatives` discipline).
**Coverage stays button-only** (don't perturb pure-explore convergence/`plan_hash`); the map is a superset
used only for authoring, accumulated in the `ground` node (dedup by `semantic_id`).

## §3. Two-phase goal — `GoalPlanner.build_scenario` (one-shot)

`GoalPlanner.build_scenario(flat_map, goal)` — **one** budget-tracked LLM call (not per-step): the LLM
returns an ordered list `{ref: <semantic_id>, verb, value?}`. **`grounding`:** a `ref` not in the map —
**or** a `verb` outside `{click,fill,type,select,press,assert}` — drops to `unmatched` (never fabricated).
**Cross-page `navigate` steps are synthesized in CODE** from the
element's `page` key (a real URL, never LLM free-text). The M9.2a per-step `propose` stays in the codebase
for the future live/co-pilot path (M9.4), but is no longer wired as the walk planner.

## §4. describe-first — `DescribePlanner.draft` + deterministic `reconcile`

`DescribePlanner(description, backend).draft()` — one LLM call → intended steps
`{verb, intent, hypothesized_target}` (ungrounded by design). `reconcile(draft, site_map)` — a
**deterministic code matcher** against the map (normalized role+name exact → text → page-scoped) → bind to
a real element **or** `unmatched`. **Conservative: never guesses** (ambiguity → unmatched). Stronger
`grounding` than goal: the LLM never picks a selector/index.

## §5. Artifacts + the determinism invariant

- **`plan.json`** (walk+scenario, single `canonical_plan_hash`, existing path) **+** a standalone
  **`scenario.json`** (the clean replayable business process — what CI runs). Steps carry `phase:
  explore|scenario`. describe additionally writes **`reconcile-report.json`**.
- **Determinism invariant:** each authored step carries the **full grounded `locator`+`alternatives`
  copied from the map element**, and **no nondeterministic field** (LLM `reason`/score) enters the step
  dict (`canonical_plan_hash` hashes everything) → the frozen plan **replays LLM-free and deterministically**.
  goal/describe are best-effort (like ADR-019), but the frozen replay is deterministic.

## §6. Exit codes

- describe with **any** unmatched step → **exit 1** (CI catches "the described flow doesn't exist").
- goal/describe producing **zero** grounded steps → **exit 1**.
- `GOAL` AND `DESCRIBE` both set → **exit 3** (mutually-exclusive heads).
- Malformed RunConfig → **exit 3** (as in M9.2a). Also: an unknown `--scenario` (no such name in
  `scenarios:`) and an entry with **both** `goal`+`describe` (XOR validation) → **exit 3** (not a silent wrong flow).

## §7. Rich RunConfig (declarative → existing env; adapters = M9.7)

```yaml
auth:                       # declarative -> M9.1 env (NO new runtime)
  storage_state: state.json #  -> STORAGE_STATE
  storage_state_save: out.json #  -> STORAGE_STATE_SAVE
  login_plan: login.json    #  -> PLAN_FILE (login-as-test)
  pw_no_trace: true         #  -> PW_NO_TRACE=1
scenarios:                  # a named list; --scenario <name> picks ONE (§C: one mode per run)
  - {name: login, goal: "log in as admin"}
  - {name: signup, describe: "register a new user"}
```

`auth`/`scenarios` join the allow-list + are validated (malformed nesting → exit 3). `_apply_auth` maps to
`STORAGE_STATE*`/`PLAN_FILE`/`PW_NO_TRACE`; `_apply_scenarios(selector)` takes the `--scenario` entry →
`GOAL`/`DESCRIBE`. Precedence **flag>file>default** via `_overridable` keyed on the target env var (a
pre-exported `STORAGE_STATE` beats the file). Budgets unchanged (vision → the `heal` role; per-role
endpoints = M9.7).

## ADR

- **ADR-028** (2026-06-27) — M9.2b two-phase authoring: goal/describe modes run a full deterministic
  heuristic explore (a site map generalized beyond buttons to input/select/link), then a **one-shot
  grounded phase-2 head** (`GoalPlanner.build_scenario` / `DescribePlanner.draft` + a deterministic
  `reconcile` in the new `brain/scenario.py`); cross-page navigates are synthesized in code; authored
  steps carry the full grounded `locator`+`alternatives` (replay is LLM-free and deterministic);
  `plan.json`(walk+scenario) + `scenario.json` + `reconcile-report.json`; a rich RunConfig (declarative
  auth/scenarios + `--scenario`). **Supersedes** the M9.2a wiring where `GoalPlanner` was the walk planner
  (`propose` retained for M9.4 live/co-pilot). describe-unmatched → exit 1; `GOAL`⊕`DESCRIBE` → exit 3.
  Accepted.
